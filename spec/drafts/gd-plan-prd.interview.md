# gd-plan-prd 인터뷰 설계 (DRAFT 메모)

> `/gd-plan-prd` 가 prd.md 를 채우기 위해 던지는 **구조화 질문 세트**. 무작위 대화가 아니라 이 순서·형식.
> **핵심 규칙 — 질문 <-> prd 필드 완전 커버:** prd.template 의 *모든 필드*는 최소 1개 질문으로 커버돼야 한다. **prd 템플릿에 필드를 추가하면 여기 질문도 추가** (둘은 항상 동기화).
> 레퍼런스: manyfast 온보딩 6문항. 우리는 톤 · roles/access · capability ID · 비기능 · Out of Scope 를 추가.

| # | 질문 | 선택 | -> prd 필드 |
|---|---|---|---|
| Q1 | 한 줄로, 무엇을 만드나? | 서술 | 한 줄 요약 |
| Q2 | (선택) 왜 지금 이걸? | 서술 | 배경/맥락 |
| Q3 | 누구를 위해? (타겟 사용자) | 복수 | 페르소나 |
| Q4 | 그들이 겪는 핵심 문제? | 복수 | (배경/핵심가치 근거) |
| Q5 | 이 제품의 핵심 목표·가치? | 단일 | 핵심 가치 |
| Q6 | (선택) 대안과 뭐가 다른가? | 서술 | 차별점 |
| Q7 | 톤·보이스는? (친근/전문/미니멀/대담/신뢰 등) | 복수 | 톤 키워드 (+ design.md 픽 힌트) |
| Q8 | 역할이 나뉘나? 각 역할이 뭘 하나? | 복수 | roles |
| Q9 | 접근 모델 + 디자인 함의? (역할기반/속성/관계) | 단일 | access model (RBAC/ABAC/ReBAC) |
| Q10 | 할 수 있어야 하는 것? (주체 역할 지정) | 복수 | capabilities `[CAP-..]` (+주체) |
| Q11 | 성공을 어떻게 측정? | 복수 | 성공 기준 |
| Q12 | 어떤 환경/기기? | 복수 | 기기/플랫폼 |
| Q13 | (선택) 비기능 요구? (성능/접근성/다국어) | 복수 | 비기능 요구 |
| Q14 | 안 하는 것? | 복수 | Out of Scope |
| Q15 | (선택) 주요 리스크? | 복수 | 리스크 |

> **커버리지 체크** — prd 필드 전부 매핑됨: 한줄 / 배경 / 페르소나 / 핵심가치 / 차별점 / 톤 / roles / access / capabilities(+주체) / 기기 / 비기능 / 성공기준 / 리스크 / Out of Scope. (빠진 필드 0)

## 진행 규칙
- 한 번에 1~2문항. 답을 prd 초안에 즉시 반영하며 진행 (멱등 — 재실행 안전).
- **capability(Q10)마다 주체 역할을 Q8 roles 에서 고르게 함** (role -> capability 연결을 인터뷰가 강제).
- Q9 access model: 대부분 RBAC. ReBAC(소유/공유) · ABAC(조건부)면 디자인 함의 1줄 추가 확인.
- 답이 추상적이면 후보를 제시(복수 선택형) — manyfast 처럼 선택지를 주어 구조화.
- 톤(Q7)은 통제 어휘에서 — design.md 픽(`/gd-plan-design`)의 힌트로도 쓰임.

## manyfast 6Q 대응
- manyfast Q1(타겟)=우리 Q3 / Q2(어려움)=Q4 / Q3(목표)=Q5 / Q4(핵심기능)=Q10 / Q5(기대경험)=Q5 / Q6(환경)=Q12.
- 우리 추가분: Q1 한줄, Q2 배경, Q6 차별점, **Q7 톤**, **Q8/Q9 roles+access**, **Q13 비기능**, Q14 Out of Scope, Q15 리스크.

## 일반 원칙 (다른 문서에도 적용)
- "질문 <-> 필드 완전 커버"는 **모든 템플릿 공통**. structure / flows / ui-rules 의 스킬 인터뷰도 각 템플릿 필드를 빠짐없이 물어야 한다. (템플릿 = 질문의 source of truth)
