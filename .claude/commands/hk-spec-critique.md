Plan Accept 전에 호출하여 spec.md의 품질을 독립 관점에서 검증합니다.
결과는 `critique.md`로 저장되며, 메인 에이전트가 검토 항목을 사용자와 함께 spec.md에 반영합니다.

## 1. 현재 Spec 확인

```bash
bash .harness-kit/bin/sdd status --json
```

`spec` 필드로 spec 디렉토리를 특정합니다. 활성 spec이 없으면 사용자에게 보고하고 멈춥니다.

`specs/<spec-dir>/spec.md`가 존재하는지 확인합니다. 없으면 멈춥니다.

## 2. Opus 서브에이전트로 비판 수행

Agent tool (`subagent_type: general-purpose`, `model: "opus"`)을 사용하여 **별도 컨텍스트**에서 비판을 수행합니다.

서브에이전트에게 전달할 프롬프트:

> 당신은 독립적인 시니어 소프트웨어 아키텍트입니다. 주어진 spec.md를 비판적으로 검토하고 개선안을 제시하세요.
>
> **입력 자료**:
> - spec 문서: `specs/<spec-dir>/spec.md`를 읽어서 요구사항을 파악하세요.
> - 필요시 `specs/<spec-dir>/plan.md`도 참조하세요.
>
> 다음 4가지 관점에서 검토하세요:
>
> ### 1. 유사 기법 조사
> 웹 검색으로 이 spec이 해결하려는 문제와 동일/유사한 문제를 해결한:
> - 알려진 디자인 패턴 또는 아키텍처 패턴
> - 오픈소스 라이브러리 또는 도구
> - 업계 모범 사례(best practice)
>
> 검색 결과와 현재 spec의 접근법을 비교하세요.
>
> ### 2. 요구사항 비판
> - **누락**: 명시되지 않았지만 필요한 요구사항이 있는가?
> - **모순**: 서로 충돌하는 요구사항이 있는가?
> - **과잉 설계(YAGNI)**: 현재 문제 범위를 벗어난 요구사항이 있는가?
> - **모호함**: 구현 시 해석이 달라질 수 있는 표현이 있는가?
>
> ### 3. 대안 제안
> 현재 접근법과 다른 방향 2~3개를 제시하세요. 각 대안에 대해:
> - 핵심 아이디어 (1~2문장)
> - 장점
> - 단점
>
> 마지막에 **권장안**을 명시하세요 (현재 spec 유지 포함 가능).
>
> ### 4. ADR 후보 추출
> 본 spec 의 결정 중 *장기 자산* 으로 박을 가치 있는 것이 있는지 식별하세요 (constitution §6.3, §6.4).
> *자동 식별 강제 X* — 검토자 판단을 돕는 체크 섹션입니다. 후보가 없으면 "없음" 으로 표기.
>
> 판단 기준:
> - cross-spec / long-lived 인가?
> - 6 개월 이상 유지될 가능성이 있는가?
> - frontmatter `type:` 어휘 (`decision` / `invariant` / `convention` / `tradeoff`) 중 하나에 해당하는가?
>
> 출력 형식 (한국어):
> ```
> # Spec Critique: <spec-id>
>
> ## 1. 유사 기법 조사
> ### 발견된 패턴/도구
> - **<이름>**: <설명> — 현재 spec과의 비교: <차이점>
>
> ### 시사점
> <현재 접근법에 반영할 수 있는 인사이트>
>
> ## 2. 요구사항 비판
> ### 누락
> - <항목>: <이유>
>
> ### 모순
> - <항목>: <이유>
>
> ### 과잉 설계
> - <항목>: <이유>
>
> ### 모호함
> - <항목>: <이유>
>
> ## 3. 대안 제안
> ### 대안 A: <이름>
> - **아이디어**: ...
> - **장점**: ...
> - **단점**: ...
>
> ### 대안 B: <이름>
> ...
>
> ## 권장안
> <현재 spec 유지 / 대안 A / 대안 B 중 선택 + 이유>
>
> ## 4. ADR 후보 추출
> - [ ] **후보 발견**: `<slug-후보>` — type: <decision / invariant / convention / tradeoff> — 이유: <한 줄>
> - [ ] **후보 없음**: <이유 한 줄>
> ```
>
> 결과는 반드시 **한국어**로 작성하세요.
> 발견된 것이 없는 항목은 "해당 없음"으로 표기하세요.

## 3. 결과 저장

비판 결과를 `specs/<spec-dir>/critique.md`에 저장합니다.

## 4. 반영 항목 선택

메인 에이전트가 `critique.md`를 읽고 반영 가능한 항목을 정리하여 사용자에게 제시합니다.

```
📋 Critique 완료: <spec-id>
권장안: <현재 유지 / 대안 X>

반영 가능한 항목 (Y/n):

[ ] 1. [누락] <항목 요약>
[ ] 2. [모호함] <항목 요약>
[ ] 3. [대안 시사점] <항목 요약>
...

→ 반영할 항목 번호를 입력하거나 "all" / "none"을 선택하세요.
  거부(n/no/아니/취소/cancel/none) 외 모든 응답 → 긍정으로 처리
```

사용자가 선택한 항목만 `spec.md`에 반영합니다.

반영 후:
```
✅ spec.md 업데이트 완료 (<N>건 반영)
→ Plan Accept 진행 가능합니다.
```

반영할 항목이 없으면 바로 Plan Accept 진행 가능합니다.
