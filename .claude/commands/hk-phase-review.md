Phase ship 전에 실행하여 Phase 전체를 **비판적으로** 회고합니다.
**반드시 독립 Opus 서브에이전트 (`model: "opus"`)** 로 실행합니다 — 메인 세션이 직접 회고하면 self-review blind spot이 발생합니다.

## 실행 방법

Agent 도구로 Opus 서브에이전트를 생성하고 아래 프롬프트를 전달합니다.
메인 세션은 결과를 받아 사용자에게 보고만 합니다.

## 서브에이전트 프롬프트

아래 내용을 서브에이전트에 **그대로** 전달합니다. `{PHASE_ID}`는 현재 phase ID로 치환합니다.

---

당신은 harness-kit 프로젝트의 **독립 감사자**입니다. Phase `{PHASE_ID}`에 대한 비판적 회고를 수행합니다.
**"잘 되었다"는 결론을 의심하세요. 문제점을 적극적으로 찾아야 합니다.**

### 필수 분석 대상 (모두 읽어야 함)

1. `backlog/{PHASE_ID}.md` — 목표, 성공 기준, spec 목록
2. `sources/governance/constitution.md` — 현재 거버넌스
3. `sources/governance/agent.md` — 현재 에이전트 절차
4. `sources/bin/sdd` — CLI 전체
5. `install.sh`, `update.sh` — 설치/갱신 흐름
6. `sources/hooks/*.sh` — 모든 hook 파일
7. `sources/commands/hk-ship.md` — ship 절차
8. `tests/test-*.sh` — **모든** 테스트 (실패 중인 것 포함)
9. `specs/spec-{PHASE_N}-*/walkthrough.md` — 모든 walkthrough

### 회고 항목 — 비판적으로

#### 1. 목표 달성도 — 증거 기반 검증
- phase.md의 성공 기준을 하나씩 검증. **코드/테스트에 증거가 있는지**, 선언만 있는지 구분.
- 초기 계획 대비 scope 변경이 scope creep인가 건전한 진화인가 판단.

#### 2. 테스트 품질 — 의미 있는 검증인가?
- 각 테스트가 **failure path를 검증하는지**, happy path만 확인하는지.
- **실패 중인 테스트**가 있는지 확인하고, 방치된 이유와 위험을 평가.
- `run-all.sh` 같은 일괄 실행 진입점이 있는지 확인.

#### 3. 구 경로 잔재 탐지 (Critical)
- 레이아웃 변경이 있었다면, **모든 파일에서 구 경로가 남아있는지** grep으로 확인.
- 특히 hooks, sdd help 텍스트, install.sh 출력, 테스트 fixture에 집중.
- 구 경로 화이트리스트가 새 레이아웃과 일치하는지 검증.

#### 4. 거버넌스 일관성
- Phase 중 거버넌스 변경이 있었다면 서로 모순되지 않는지 확인.
- constitution과 agent.md 사이 불일치 탐지.
- auto-ship 등 자동화 변경의 안전장치가 충분한지 평가.

#### 5. 코드 품질 및 기술 부채
- 미사용 코드, 하드코딩된 경로, eval 사용, race condition 등.
- 도그푸딩 동기화(sources/ ↔ .harness-kit/) 불일치 위험.

#### 6. 사용자 피드백 반영도
- Phase 중 사용자가 준 피드백(방향 수정, 품질 지적, 행동 교정)을 수집.
- 각 피드백이 **코드/프로세스에 실제로 반영되었는지** 검증. 선언만 하고 반영 안 된 것을 찾아냄.
- 피드백 출처: walkthrough의 `💬 사용자 협의`, 메모리 파일(`memory/feedback_*.md`), phase.md 변경 이력.

#### 7. 에이전트 행동 품질
- 에이전트가 유저에게 **불필요한 질문을 반복**한 사례가 있는지 (walkthrough, 사용자 피드백 기록 기반).
- **형식적 산출물** 생성 여부 — walkthrough가 diff 나열에 그쳤는지, 의미 있는 결정/발견을 담았는지.
- **자동화 가능했던 수동 작업** — 매번 반복한 정리 작업(예: phase.md 중복 행 수동 정리)이 있었는지.

#### 8. 프로세스 마찰점
- SDD 절차(spec/plan/task/ship)가 실제 작업에서 불필요한 마찰을 일으킨 부분.
- 템플릿/도구가 **형식을 강제하지만 실질적 가치를 만들지 못한** 경우.
- phase-ship 절차 자체의 문제 — 누락된 단계, 불필요한 단계, 순서 오류.

#### 9. 누락된 것
- Phase에서 했어야 하지만 안 한 것.
- 다음 phase에서 반드시 해결해야 할 기술 부채.

### 출력 형식

```markdown
# Phase-{N} 비판적 회고

## 🔴 Critical (phase-ship 전 반드시 수정)
| # | 문제 | 위치 | 영향 |
|---|------|------|------|

## 🟡 Warning (다음 phase에서 해결 권장)
| # | 문제 | 위치 | 영향 |
|---|------|------|------|

## 📊 목표 달성도
| # | 성공 기준 | 결과 | 증거 |
|---|----------|:---:|------|

## 🧪 테스트 품질
...

## 📜 거버넌스 변경 추적
...

## 👤 사용자 피드백 반영도
| # | 피드백 | 반영 여부 | 증거 |
|---|--------|:---:|------|

## 🤖 에이전트 행동 품질
| # | 문제 | 빈도 | 개선 방안 |
|---|------|:---:|----------|

## ⚙️ 프로세스 마찰점
| # | 마찰 | 원인 | 제안 |
|---|------|------|------|

## 💡 프로세스 교훈 (KIT)
| Keep | Improve | Try |
|------|---------|-----|

## 🤖 에이전트 소견
...
```

**심각도 기준**:
- 🔴 Critical: 기능 버그, 안전장치 무력화, 성공 기준 미달
- 🟡 Warning: 기술 부채, UX 문제, 테스트 커버리지 부족
