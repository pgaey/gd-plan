spec-13-01 · 딥다이브

문서를 "규칙"으로 + 근육(컴파일러) 아키텍처

두 가지를 다룹니다. (1) "기획 문서 5개"를 추상적 글쓰기가 아니라 검사 가능한 규칙 체계로 만드는 법(추천), (2) 기존 gen-design의 "근육" — chat.md → React 컴파일러가 실제로 어떻게 생겼고 왜 studio에 "박혀" 있는지(정말 친절하게).

Part 1 · 문서를 규칙으로 연결된 모델 5종 규칙 추천 Part 2 · 근육 아키텍처 "박혔다"의 뜻 컴파일 파이프라인 핵심 자산 5
PART 1

"문서 5개 만들기"를 규칙으로 바꾸기

맞는 직감입니다 — "prd/design/structure/flows/ui-rules를 쓴다"는 말은 너무 추상적입니다. 자유 작문이면 매번 다른 결과가 나오고, 하류(/gd-chat)가 활용하기도 어렵습니다. 핵심은 각 문서를 "무엇을·어떤 규칙으로" 채울지 정하는 것.

핵심
5종 문서를 따로 노는 자유 글 5개가 아니라 — ID로 서로 연결된 하나의 모델로 보세요. 그리고 그 내용은 하류가 실제로 소비할 수 있는 어휘(컴포넌트 레지스트리·토큰)에 묶으세요. 그러면 (a) 검사가 가능해지고 (b) /gd-chat 주입이 정확해집니다.

① 5종은 "ID로 연결된 모델"이다

각 항목에 안정적인 ID를 달면 "예약하기(CAP-01)는 어느 page가 담당? 그 page는 어느 flow에 나옴? 그게 어느 chat.md scene이 됨?"이 기계적으로 추적됩니다. 끊긴 고리(예: 후기쓰기를 담당할 page 없음)는 자동으로 잡힙니다. → 이게 /gd-plan-review의 BLOCK이고, v2의 "결정적 판정"의 토대.

② 문서별 추천 — 구조 / 핵심 규칙 / 하류 연결

prd.md

무엇을 만드나

구조한 줄 요약 · 페르소나(1–3) · capability 목록 [CAP-NN] · 성공기준 · 비범위 · 톤 키워드
규칙capability는 ID + 동사로 시작("예약하기"). 톤 키워드는 통제된 어휘에서.
하류모든 CAP은 structure의 page로 매핑돼야(BLOCK). 톤 → narrative.
design.md

어떤 톤·비주얼인가

구조collection 픽 + frontmatter(pickedFrom/reason) + 톤·색·타이포 요약
규칙0부터 안 만들고 collection에서 픽. 톤 키워드 = 통제 어휘.
하류톤 → narrative · 색/radius → tokens · 컴포넌트 스타일 → 레지스트리.
structure.md

어떤 페이지·구조인가

구조sitemap [PAGE-slug] · 페이지별 섹션 목록 · 레이아웃(LNB/Sticky/Modal) · 반응형 · 상태(empty/loading/error) · covers:[CAP..]
규칙page에 ID. 섹션은 통제된 섹션 어휘. 각 page가 담당하는 capability 명시.
하류page ↔ scene(chat.md) 매핑 · 섹션 순서 → chat.md Structure 트리 순서.
flows/*.md

사용자 여정은

구조이름·목적·actor·trigger · steps[→PAGE-id] · edge cases · success
규칙step이 참조하는 page는 sitemap에 반드시 존재. step마다 페이지·컴포넌트·데이터 명시.
하류scene 진입/이탈 컨텍스트 → chat.md narrative/history.
ui-rules.md

수치 규칙은

구조spacing · radius · motion · header · CTA 위계 · form · mobile · a11y — 전부 수치
규칙수치는 가능하면 token으로 표현(또는 token과 일치). design.md와 충돌 금지.
하류수치 → tokens.json → 컴파일러가 적용. ui-rules ↔ tokens 1:1이 이상적.
정리
추천 한 줄: "자유 서술"을 버리고 "통제된 어휘 + 안정 ID + 하류(token·레지스트리) 매핑"을 가진 구조화된 템플릿으로. 이러면 "문서 5개"가 추상에서 벗어나 채우면 곧 검사·주입 가능한 모델이 됩니다. (단, ID 체계의 "결정적 검사"까지는 v2 — v1은 템플릿 구조 + LLM 판정으로 시작 가능)
PART 2

근육 = chat.md → React 컴파일러

"근육"은 chat.md(화면 묘사 텍스트)를 실제 React 코드로 바꾸는 결정적 변환기입니다. 두뇌(대화 스킬)와 달리, 이건 코드예요. 그리고 이 코드가 studio 안에 "박혀" 있습니다.

① "studio에 박혔다"가 무슨 뜻?

컴파일러가 독립된 패키지가 아니라, studio라는 앱의 소스 폴더(studio/src/lib/…) 안에 파일로 들어 있다는 뜻입니다. 그래서 컴파일러만 따로 npm install 할 수 없고, studio 소스가 옆에 있어야만 돕니다. gd-cli는 긴 상대경로로 studio 안으로 "손을 뻗어" 끌어다 씁니다.

비유: 컴파일러가 특정 차(studio)에 용접된 엔진 같은 상태. 성능은 검증됐지만 다른 차(gd-plan 기반 새 토대)로 옮기려면 먼저 용접을 떼어 독립 엔진 블록(@gd/chat-md-core)으로 만들어야 합니다. 그 떼어내기 설계가 바로 spec-12-07(pluginarch)인데 아직 미구현입니다.

② 컴파일 파이프라인 — chat.md가 React가 되기까지

  1. chat.md 읽기 — 디자이너가 만든 화면 묘사 텍스트(3-layer). studio/src/lib/chat-md-compiler/react/compile-scene.ts
  2. 파싱(peggy 문법) — 텍스트를 문법 규칙으로 해석. studio/src/lib/chat-md/grammar/chat-md.ts (peggy, 488줄)
  3. AST 생성 — <Card><Button>… 같은 트리(컴포넌트·placeholder·토큰)로 변환. chat-md/parser/ast-types.ts
  4. shell 합성 — 전역 외각(_shell.chat.md)의 {{scene.content}} 자리에 이 화면을 끼움. react/shell-merge.ts
  5. 섹션 분리 + JSX emit — Behavior(useState)·Variants 분리 후 AST를 JSX 문자열로. placeholder는 {t("…")}/{tokens["…"]}로. react/jsx-emitter.ts · compile.ts
  6. import 연결 — 쓴 컴포넌트 이름을 레지스트리에서 찾아 @/components/… import 구문 생성. react/imports-builder.ts
  7. (선택) order.md 주입 — <slug>.order.md가 있으면 zod 스키마·useForm·useQuery(TanStack)·Skeleton을 TSX에 끼움. packages/gd-cli/src/commands/order.ts
  8. 단일 React TSX 출력 — export function <Name>(){ return (…) }. 맨 앞에 // @gd: … 주석(drift 추적용).

③ 새로 짜든 떼어내든, 반드시 이해해야 할 핵심 자산 5

peggy 문법 — chat.md의 전체 규칙(frontmatter + 3-layer + 컴포넌트 태그 + placeholder)이 한 파일에 집약. 재현 비용이 가장 큼.studio/src/lib/chat-md/grammar/chat-md.ts
AST 타입 — 문법과 모든 emitter의 "계약". 4-layer 어휘(props/theme/tokens).studio/src/lib/chat-md/parser/ast-types.ts
react codegen — shell 합성 + 결정적 JSX emit 로직.studio/src/lib/chat-md-compiler/react/ (compile-scene·compile·jsx-emitter·shell-merge·imports-builder)
컴포넌트 레지스트리 — 어휘 SSOT(이름→import 경로). 현재 paper/ 안에 있어 "react가 paper에 의존"하는 결합의 원인.studio/src/lib/chat-md-compiler/paper/component-registry-metadata.ts
order.md → 동작 코드 — 정적 UI 너머(폼 검증·데이터 fetch)를 코드로. chat.md(구조)와 order.md(동작)의 2-파일 분리.packages/gd-cli/src/commands/order.ts
결합 요약
gd-cli가 react/lint/diff/paper-import 전부 ../../../../studio/src/lib/…로 studio를 직접 참조 + 빌드 없이 tsx로 실행 → 단독 배포 불가. 떼어내기 설계(=@gd/chat-md-core 추출, 레지스트리 DI)는 spec-12-07에 있지만 미구현(보류). "벗어나기"의 실제 난이도는 전부 여기에 있습니다.
