한눈 해석 — 3가지 성격

Google (getshipit)

단일 종합 PRD. 시장·비즈니스 모델·가격·기술 명세까지 — PM 전반을 한 문서에. 무겁지만 빠짐없음.

manyfast

lean + 구조화 + 멀티 문서. 트리 5섹션, 6문항 온보딩으로 생성. 기능명세서는 분리. 속성(roles/기기)을 구조화 필드로.

우리 (gd-plan)

lean + 구조화 + 멀티 문서. 디자인은 design/structure/ui-rules 로 분리. capability 에 ID. manyfast 와 같은 계열.

핵심
manyfast 가 우리 접근을 검증합니다 — 둘 다 "lean PRD + 구조화 필드 + 멀티 문서(기능/구조 분리) + 인터뷰로 생성". 즉 Google 식 비즈니스 무게는 우리 범위 밖이 맞고, 우리가 더 배울 곳은 manyfast 입니다.

추가 검토 영역 (우리 prd 에 넣을 빈자리)

manyfast 에는 있고 우리에 없는 것 중, gd-plan 의 "통제" 목적에 도움되는 것:

  • 차별점 (differentiation) 추천 — "대안과 무엇이 다른가". 셋 중 manyfast 만 명시. 핵심가치와 별개로 USP 를 또렷하게.
  • 사용자 역할(roles) 추천 — User/Admin 같은 역할을 구조화 필드로. manyfast 는 이걸 기능명세와 연결 → 우리도 capability 의 주체를 명시(누가 이 capability 를 쓰나). structure 의 page 권한과도 연결.
  • 리스크 추천 — 가벼운 리스크 목록. 지금은 phase 차원에만 있고 프로젝트 PRD 엔 없음.
  • 배경/맥락 선택 — "왜 지금 이걸". 한 줄 요약 위에 짧게.
  • 타겟 기기/플랫폼 선택 — Web/Mobile 범위. 우리는 React web 전제라 암시적이지만 명시하면 responsive 범위가 분명.
  • 대표 시나리오 선택 — manyfast 는 PRD 안에 1개. 우리는 flows 로 분리하니 중복일 수 있음 → PRD 엔 "대표 1줄"만 옵션.
범위 밖
Google 의 비즈니스 모델/가격·마케팅·정량 고객데이터·기술 명세는 gd-plan(디자인 상류) 범위 밖. 넣지 않음. 디자인/UX 는 우리 design/structure/ui-rules 가 담당(빈자리 아님).

보너스 — manyfast 온보딩 6문항 ≈ 우리 gd-plan-prd 인터뷰

manyfast 는 6개 구조화 질문으로 PRD 를 채웁니다. 우리 /gd-plan-prd 인터뷰 설계의 좋은 레퍼런스:

  • Q1 주요 타겟 사용자 (복수) → 페르소나/역할
  • Q2 주요 어려움 (복수) → 사용자 문제
  • Q3 궁극 목표 (단일) → 제품 목표
  • Q4 핵심 기능/접근 (단일) → 차별점/핵심 capability
  • Q5 기대 경험 (단일) → 가치/성공 기준
  • Q6 사용 환경 (단일) → 기기/플랫폼
